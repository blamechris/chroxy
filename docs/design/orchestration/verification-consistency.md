# Cross-Slice Adversarial Review — Orchestration Design (engine / metering / surface)

## 1. CONTRADICTIONS

**F1. Run-state enum: engine states ≠ wire `RUN_STATUSES`.** Engine §3.2 defines `created, planning, plan_review, executing, synthesizing, cancelling, cancelled, suspended, budget_paused, completed, failed`. Surface §1.4 defines `RUN_STATUSES = ['created','planning','awaiting_gate','delegating','reviewing','merging','budget_paused','completed','failed','cancelled']`. Surface §6.2 requires the engine to *import* these enums, but `plan_review`, `synthesizing`, `cancelling`, and (critically) `suspended` have no wire representation — a suspended-after-restart run cannot serialize into `RunSummarySchema.status` (strict `z.enum`, `safeParse` drop). Also `awaiting_gate/delegating/reviewing/merging` don't exist in the engine machine. **Resolution:** surface PR-1 adopts the engine's state list verbatim (plus keep `budget_paused`); delete the invented wire-only states. Engine §3.2 is the single source; protocol file exports it.

**F2. Subtask-state enum: engine §3.3 ≠ `RUN_NODE_STATUSES`.** Engine: `pending, spawning, briefing, poa_review, executing, result_review, respawning, merging, conflict_fixup, escalated, done, skipped, failed, cancelled, interrupted`. Surface: `['pending','planning','awaiting_plan_review','executing','awaiting_result_review','revising','merged','done','failed','cancelled']`. Missing on the wire: `spawning/respawning/merging/conflict_fixup/escalated/skipped/interrupted` — the dashboard literally cannot render the escalation state its own GateBanner UX depends on. `revising` and terminal `merged` don't exist in the engine (revise loops to `briefing`/`executing`; merge success → `done`). **Resolution:** same as F1 — wire enum = engine enum, one exporter.

**F3. Gate/escalation verb model is three-way inconsistent.** Engine: `approvePlan/rejectPlan`; escalations resolve with `accept-as-is | retry-with-note | skip | fail-run` (§3.3); budget-paused resolves with `raise / synthesize / cancel` (§11); permission escalations via `resolveGate`. Surface: one `OrchestrationGateResponseSchema` with `decision: enum(['approve','reject','revise'])` and `RUN_GATE_KINDS = ['epic_plan','worker_plan','worker_result','budget_overrun','merge']` — **no kind for subtask escalation or Bash permission escalation** (engine gates #2 and #3), while `worker_plan/worker_result/merge` are architect-internal reviews the engine never exposes as user gates. `skip` and `synthesize-now` are unrepresentable; `raise` carries no amount field. **Resolution:** gate kinds → `['epic_plan','escalation','bash_permission','budget_overrun']`; decision enum → `['approve','reject','revise','skip']` + optional `budgetUsd` (for approve-with-raise on `budget_overrun`); per-kind decision validity enforced server-side. Map: accept-as-is→approve, retry-with-note→revise, fail-run→reject, skip→skip; raise→approve+budgetUsd, synthesize→revise (semantics documented), cancel→reject. Drop architect-review "gates" (see F23).

**F4. Two incompatible run stores.** Engine §1.2/§12: `createRunStore({filePath: ~/.chroxy/orchestration-runs.json})` single file, interface `loadAll/get/upsert/appendTimeline/recordUsage/flushSync`. Metering §2: `RunLedger({baseDir})` → `~/.chroxy/orchestration/{runs-index.json, runs/<runId>/{run.json,events.jsonl,report.*}}`, interface `createRun/setStatus/createSubtask/attachSession/recordTurnUsage/.../recoverRuns`. Different path, different granularity, different recovery model (`loadAll` vs journal-replay `recoverRuns`), different runId format (hex vs `run_<ts>_<rand>`), and engine hardcodes `homedir()` while metering honors `CHROXY_CONFIG_DIR`. **Resolution:** RunLedger wins (journal+snapshot+durable-report is strictly better); engine §1.2/§12 rewritten to consume the RunLedger API — `upsert`→`setStatus`+`createSubtask`+granular mutators, `recordUsage`→`recordTurnUsage`, boot reconcile = `ledger.recoverRuns()` then engine decides suspend/fail. Delete `run-store.js`.

**F5. Directory collision → engine orphan sweep can delete the ledger.** Engine §7/§9 puts the integration worktree at `~/.chroxy/orchestration/<runId>/integration` and its reconcile removes "`~/.chroxy/orchestration/<runId>/` dirs with no active run". Metering §2.2 puts run records at `~/.chroxy/orchestration/runs/<runId>/`. The sweep's "is this dir an active runId" check would see `runs/` (and any stray file) as an orphan run and `rm -rf` the entire ledger. **Resolution:** worktrees move to `~/.chroxy/orchestration/worktrees/<runId>/`; sweep scans only that subtree; ledger owns `runs/` exclusively.

**F6. Three event vocabularies, no translator.** Engine §12 emits 7 in-process names (`orchestration_run_updated, orchestration_subtask_updated, orchestration_committee_event, orchestration_usage, orchestration_budget, orchestration_gate_escalation, orchestration_run_snapshot`). Metering §2.5 suggests wire names `orch_run_usage/orch_budget_warning/orch_budget_cap/orch_run_updated/orch_run_report`. Surface defines exactly 4 wire types, of which the only push is `orchestration_run_delta`. None of the three lists match. Worse, engine §12 wires `on('orchestration_event') → wsServer?.broadcast(evt)` — raw internal payloads, **no unbound-client predicate**, broadcast to mobile-bound clients too, violating dashboard-only and guard-2. **Resolution:** all in-process events funnel through one projection layer (see F13) into `orchestration_run_delta` via `_broadcastOrchestrationDelta` (unbound-only). Metering's `orch_*` names are dropped; budget warn/cap surface as delta `run` upsert + a timeline entry (kind `budget_warning`).

**F7. Metering prescribes dispatch-table entries; surface forbids them.** Metering §2.5: ledger wire events "need `orchestration.ts` schemas + **dispatch-table entries**". Surface §1.6 explicitly rejects dispatch-table (a dispatch-table type is credited to BOTH clients by guard-2, breaking `DASHBOARD_ONLY`'s self-check and locked decision 1). **Resolution:** surface wins — dashboard `HANDLERS` + `DASHBOARD_ONLY`/`PLATFORM_SPECIFIC` allowlists; strike the dispatch-table sentence from metering.

**F8. Budget config shape three ways.** Engine §2: flat `defaultBudgetUsd` + `budgetWarnRatio: 0.8`; run.budget `{capUsd, warnRatio, spentUsd, estimatedUsd, warned, exceeded}`. Metering §3: `defaultBudget: {maxUsd, maxTokens, warnPercent: 80, perRole:{...}}`, latched `budgetState {warnedAt, capReachedAt, capLiftedAt, perRole}`, `level: ok|warned|capped`. Surface: `RunBudgetSchema {capUsd, spentUsd, state: 'ok'|'warning'|'exceeded'}` and start message carries only `budgetUsd`. Three state spellings (`warned/exceeded` bools vs `warned/capped` vs `warning/exceeded`), two config keys, and per-role/maxTokens exist only in metering. **Resolution:** metering's evaluator is authoritative internally; v1 config = `{maxUsd, warnPercent}` only (cut `maxTokens`/`perRole` to fast-follow, F21); wire enum standardizes on `['ok','warned','capped']`; engine's `budget` block replaced by ledger's `budgetState` + `BudgetEval`.

**F9. Usage-row shape: engine §11 vs metering `turn_usage` journal line.** Engine: camelCase tokens, `turnLabel` ('plan'|'poa'|'poa_review'|…), `role: 'architect'|'worker'|'fixup'`, `costUsd + estimated: bool`. Metering: snake_case tokens, **no turnLabel at all**, dotted roles (`worker.audit`, `architect.review`), `cost` vs `pricedCostUsd` split with the provenance rule "derived cost NEVER contaminates costUsd" — which engine's `estimated:true`-flagged `costUsd` violates. Engine §11 also says "the orchestrator computes its own row" while metering §2.5 has `recordTurnUsage(runId, {sessionId, terminalEvent, data})` doing the fold+pricing inside the ledger — **two owners of cost computation**. **Resolution:** metering's line is canonical; add `turnLabel: string` to it (engine passes it in `recordTurnUsage` opts — it's the per-gate attribution the whole dogfood needs); engine does NOT compute costs — it forwards the raw terminal payload + `{role, turnLabel, subtaskId}` and the ledger prices. One role taxonomy: dotted strings (`architect`, `architect.review`, `worker.audit`, `worker.implement`, `worker.fixup`), engine's flat enum deleted.

**F10. Committee verdict enums.** Engine result-review verdicts: `accept|revise|redelegate|escalate`; poa-review: `approve|revise|redelegate`. Metering §2.3 `committee[].verdict: "approve"|"revise"|"redelegate"` — cannot record `accept` or `escalate`. **Resolution:** normalize the engine to `approve` at both gates (rename `accept`) and extend metering's enum with `escalate`.

**F11. `orchestration_run_snapshot` name collision, push vs pull.** Engine §12 lists it among *emitted broadcast events*; surface defines it as the *reply* to `orchestration_run_detail_request`. **Resolution:** it's a pull-reply only (surface semantics); engine's push need is served by deltas.

**F12. Run-action verbs without engine APIs.** Surface `orchestration_run_action: enum(['cancel','pause','resume'])`; engine's public API has `cancelRun/resumeRun` but **no pause**, and `resume` is overloaded (suspended-after-restart vs budget_paused vs user-pause). **Resolution:** add `pauseRun(runId)` to engine API (semantics = block new delegations, same as budget-pause path); document `resumeRun` as valid from `suspended|budget_paused|paused` with per-state guards.

## 2. GAPS / UNOWNED WORK

**F13. Engine-record → wire-schema projection is unowned.** Engine records (§3.1/§3.3: `subtaskId/state/goal/committee.history`) vs wire (`nodeId/status/epicPrompt/gates/timeline/usageRollup`, `RunSummary.nodeCounts/pendingUserGates/epicPromptPreview`) share almost no field names. Surface PR-2 stubs `ctx.services.orchestrator.{listRuns,getRun,...}` implying wire-shaped returns; engine never specifies producing them. **Resolution:** new engine-owned module `orchestration/to-wire.js` (pure: RunRecord→RunSummary/RunDetail, SubtaskRecord→RunNode, gate registry→RunGate, journal/timeline→RunTimelineEntry), unit-tested against the protocol schemas; deliverable of engine P6.

**F14. Wire delta `seq` assignment is unowned — and journal seq cannot be it.** Surface's client contract is strict `seq === held.seq + 1`. Metering's journal seq advances on *every* journal event (including never-broadcast `turn_usage` folds and committee bodies) and usage deltas are coalesced ≤1/s — reusing journal seq guarantees perpetual gaps → the "silently re-request snapshot" path degenerates into polling. **Resolution:** a dedicated per-run **wire seq counter**, incremented only when a delta is actually broadcast, owned by the emission layer in `orchestration-manager.js` (alongside F13's projection); snapshot replies stamp the current wire seq. State it explicitly in engine P6 scope.

**F15. `session_list.orchestrationRunId/Role` population — mutual finger-pointing.** Engine §15.3 flags it "to that [protocol] agent"; surface PR-4 says "server population from the engine". No mechanism exists: session-list entries are built by SessionManager/ws-server, which know nothing of runs. **Resolution:** engine owns it — either (a) a `metadata` opt on `createSession` persisted on the entry and surfaced by the session-list builder, or (b) ws-server consults `orchestrationManager.roleForSession(sessionId)` when building the list. Pick (a) (survives restart via session-state); assign to engine P6 + surface PR-4 together.

**F16. `orch_attach_baseline`, verdict-quality `note()`, and report retrieval have no wire path.** Metering §5 requires client→server baseline attach, a dashboard-prompted `verdictQuality` note at finalize, and report fetch (`orch_run_report` or HTTP). Surface defined 5 client types and 4 server types — none of these. The HTTP export endpoint is explicitly deferred. Net: **the dogfood measurement story is unreachable from any client in v1.** **Resolution:** add to surface PR-1 (avoid a second guard-atomicity dance): client `orchestration_run_annotate {runId, baselineSessionId?, verdictQuality?}` and fold `report` (json + markdown strings) into `ServerOrchestrationRunSnapshotSchema` as optional fields present at terminal state. Defer the HTTP export as planned.

**F17. Unified gate registry is unowned.** Surface requires stable `gateId`s, a `gates[]` array, statuses incl. `expired`, `pendingUserGates` counts. Engine has three disjoint mechanisms (`approvePlan/rejectPlan`, `resolveEscalation(runId, subtaskId,…)`, `resolveGate(runId, gateId,…)`) and no gate objects for plan-review or escalations. The handler mapping `orchestration_gate_response{gateId}` → three engine APIs is specified nowhere. **Resolution:** engine adds a first-class gate registry (`gateId`, kind, subtaskId|null, status, timestamps) in `run-model.js`; `approvePlan/rejectPlan/resolveEscalation` become internal aliases of one `resolveGate(runId, gateId, {decision, note, budgetUsd})`; permission-gate timeout resolves the gate `expired`/`resolvedBy:'policy'`.

**F18. Integration-worktree cleanup on restart-reconcile is under-specified.** Engine §9 removes it at cancel/terminal, but: (i) a run failed *during* reconcile — who removes it; (ii) post-crash `rm -rf` of the dir leaves stale `.git/worktrees/` metadata in the **user's repo** (needs `git worktree remove --force` first, then `git worktree prune -C <run.cwd>` as fallback); (iii) F5's sweep collision. **Resolution:** reconcile rule — for every run transitioning to terminal at boot, run `git worktree remove --force` / `prune` against `run.cwd`; `suspended` runs keep the worktree; document in engine §9 row "Orphaned sessions/worktrees".

**F19. Budget raise outside a `budget_overrun` gate has no wire path.** Engine API has `setRunBudget(runId, capUsd)` callable anytime; metering `setBudget` supports mid-run raise/lower with `capLiftedAt` history; surface has **no client message** for it (gate response only fires when a gate is open). **Resolution:** either add `budgetUsd?` to `orchestration_run_action` with action `set_budget`, or (leaner, recommended for v1) document budget changes as possible *only* via the `budget_overrun` gate (approve+budgetUsd per F3) and drop the free-standing `setRunBudget` from the v1 wire.

**F20. Run-level usage schema can't carry metering's honesty fields.** `RunUsageSchema = CumulativeUsageSchema` (a closed session-scoped object per metering §0) can't express `effectiveUsd` vs provider `costUsd`, `unknownCostTurns`, `meteringGaps` — so the dashboard cannot render the "observed spend ≥ shown" caveat both other slices mandate. **Resolution:** dedicated `RunUsageSchema` in `orchestration.ts` mirroring metering's UsageCell (tokens + costUsd + pricedCostUsd + effectiveUsd + unknownCostTurns), plus `meteringGaps: string[]` on `RunDetailSchema`.

**F21. Who owns `handlers/orchestration-handlers.js` / `ws-message-handlers.js`?** Both engine §1 and surface §1.5/PR-2 create the same files. **Resolution:** surface PR-2 owns them (handlers + gating + registration, driven against a stub service); engine P6 only implements the `ctx.services.orchestrationManager` interface. Engine's "Modified files" table drops both rows.

**F22. Budget check before architect/committee turns.** Engine §11 wants "no new committee cycles" at cap; metering's evaluation points are only (a) pre-`createSession` and (b) post-fold. Committee turns spawn no session, so nothing gates them. **Resolution:** engine calls `evaluateBudget(runId, {role:'architect.review'})` before every committee turn; one sentence in metering §3 evaluation points.

## 4. DESIGN SMELLS

**F23. Architect-review "gates" on the wire (surface `worker_plan/worker_result/merge`, `approver:'architect'`).** Render-only, never actionable, duplicate the timeline, and inflate the schema/gate registry for zero v1 value. Cut per F3; committee reviews are timeline entries.

**F24. Engine risk-7's "unpriced tokens > 20% of run volume" warning** — invented threshold, "volume" undefined; metering's `unknownCostTurns`/`meteringGaps` already covers it. Drop the 20% rule.

**F25. Per-role budget caps + `maxTokens` in v1 (metering §3)** — no other slice can set, display, or act on them (no wire field, no engine config, no UI). Keep the `perRole` slot in the stored shape if cheap, but cut from v1 config/eval to avoid dead half-features.

**F26. Engine §12 lazy `wsServer?.broadcast(evt)`** — even after F6's fix, note the risk: events emitted before wsServer exists are silently dropped; deltas must bump the wire seq only on *successful* broadcast or snapshots will claim seqs clients never saw. One-line spec addition in engine P6.

**F27. Gate-timeout double-answer (engine §8):** the gate "re-answers deny at timeout" after PermissionManager's own 5-min timeout has already resolved the request — the second `respondToPermission` on an expired requestId will error. Specify: check request liveness before re-answering; gate resolution → `expired` (F17) either way.

**F28. Decision-contract brace-scan fallback (engine §5.2 step 1, last balanced `{}` in final 32KB)** can grab JSON the worker merely quoted (e.g. a config sample at message end). Acceptable given the `kind` check, but add a negative fixture (quoted-JSON tail with wrong/absent kind → repair prompt, not acceptance) to P1 tests.

**F29. `estimatedUsd` on engine's RunRecord.budget** duplicates ledger-computed `pricedCostUsd`/`effectiveUsd` — after F4 the engine should hold **no** money math; budget display state comes exclusively from `BudgetEval`. Delete the field.

**F30. runId format**: pick metering's `run_<ts>_<rand>` (sortable, self-describing); engine §3.1 updated.

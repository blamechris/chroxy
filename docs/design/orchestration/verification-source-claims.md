**1. permission_request re-emit + respondToPermission across providers — CONFIRMED, with critical gaps.**
- Re-emit: `session-manager.js:2884` lists `permission_request` in `builtinTransient`; forwarded at `:2889` via `this.emit('session_event', { sessionId, event, data })`. PermissionManager is the emit source (`permission-manager.js:295`).
- `respondToPermission(requestId, decision, editedInput)`: exists on **sdk-session.js:1295-1296**, **codex-app-server-session.js:508-510** (`inProcessPermissions:true` at `:67`), **byok-session.js:1733-1752**. All delegate to `_permissions.respondToPermission`.
- **cli-session.js: NO respondToPermission.** `permissions:true` but `inProcessPermissions:false` (`:237-238`). Permissions are answered out-of-band via an HTTP **permission hook** (`createPermissionHookManager` `:417`, `permission-hook.js`, POST `/permission` `:495`), NOT in-process. An in-process engine cannot answer a cli worker with `entry.session.respondToPermission`.
- **gemini-session.js: NO permission surface at all.** `permissions:false, inProcessPermissions:false, permissionModeSwitch:false` (`:172-175`). It emits `tool_start`/`tool_result` only — never `permission_request`, no respond method.
- Consequence: in-process gate works for sdk/codex/byok; cli requires driving the hook; gemini cannot be gated.

- Consequence: strict read-only audit workers must be Claude (sdk/byok) or codex with the sandbox resolver forced to `read-only`; **gemini audit workers cannot be constrained** to read-only through any Chroxy surface.

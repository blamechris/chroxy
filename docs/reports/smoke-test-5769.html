Why a human has to drive this

This PR refactors the most fragile, empirically-pinned code in the repo — the AskUserQuestion form driving that types keystrokes into a live claude TUI. The change is a pure receiver swap (mixin → injected collaborator) and is proven byte-for-byte behavior-preserving by construction + 6 new isolated unit tests + 340 existing tests + Copilot.

But project memory (tui_multi_question_form_keys, multi_question_post_deny_wedge, #4668) is blunt: some form-driving edge cases only ever surface against a real PTY. So before we merge, we drive a couple of real AskUserQuestion forms through a live session and watch the TUI actually land on the option you picked. ~10 minutes.

The keystroke logic did not change in this PR — so if forms behaved correctly on main, they must here. Any misbehavior you see is a real regression and a hard STOP on the merge.

1Start the branch server on a throwaway port (loopback only)

This runs this branch's source (not the published npx chroxy, not your :8765 daemon) with the claude-tui provider, no tunnel, no auth, bound to loopback, on port 8799. Run it in its own terminal and leave it streaming logs — you'll watch them during the tests.

PORT=8799 PATH="/opt/homebrew/opt/node@22/bin:$PATH" \
  node packages/server/src/cli.js start \
  --provider claude-tui \
  --no-auth \
  --no-supervisor \
  --host 127.0.0.1 \
  --skip-checks \
  --dangerously-skip-permissions \
  --log-format text

--no-auth disables the tunnel and token (loopback dev only). --no-supervisor keeps it a single foreground process so Ctrl-C cleanly stops it. --dangerously-skip-permissions stops chroxy's permission gate from interrupting the form flow — it does not affect FormDriver, it just keeps the run smooth. PORT=8799 guarantees no collision with the :8765 daemon.

If the dashboard looks stale you can optionally rebuild it first (FormDriver is server-side, so this is rarely needed): npm run build -w @chroxy/dashboard

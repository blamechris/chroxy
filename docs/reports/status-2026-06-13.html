Design decisions taken on your behalf

#5697 — closed wontfix, not "fixed for parity." The proposed guard (force a fresh response bubble whenever a live stream_start collides with a prior response id) is defensive against a bug the swarm confirmed is unreachable (server response ids are monotonic + per-boot-prefixed). Implementing it (PR #5723) failed CI because both clients intentionally reuse a response bubble on a live same-id stream_start (the post-stream_end continuation path, encoded in connection.test.ts). Changing core streaming behavior to defend an impossible collision fails the "do no harm" bar — so it was reverted and the issue closed with the trigger-to-revisit recorded.
#5698 reconnect — client cap first; server-lifecycle change deferred. The biggest decision here is whether the supervisor should park in a terminal "I'm down" standby (serving {status:'down'}) instead of _exit(1) on max-restarts. That changes process-exit semantics under launchd/the desktop app — real blast radius. Decision: ship the pure-client ladder cap first (zero server risk, fixes the core "spins forever" jank), and defer the server-side positive-down signal (FIX-1) until the lifecycle question can be made deliberately. The client cap alone takes the user from an infinite spinner to a clear "Server appears to be down — Reconnect."
Scope discipline. #5698 was split: FIX-2 (client cap, dashboard) shipped now; FIX-1 (server signal), FIX-3 (per-session terminal flag), and the mobile-app wiring + resume/network clear are tracked follow-ups (#5725 for mobile; FIX-1/FIX-3 under the open parent #5698). The dashboard is "the app" you use day-to-day, so it got the surface first.

Process lessons banked (project memory)

  • Review agents must be worktree-isolated — a non-isolated review agent ran git checkout and switched the working tree mid-edit. All review agents now run in a worktree or are forbidden from switching branches.
  • Run the full package suite before pushing shared store-core changes — a targeted-file-green change went red on contract/integration tests in CI; the shared module ripples widely.

4 · Harness "init instructions" — your question, answered

You asked whether chroxy can prepend/send initialization instructions to whichever model a session uses, telling the agent about the harness and how to leverage it. The plumbing already exists; the convenience layer doesn't.

What exists today (system-prompt injection)

  • Per-session preamble — sessionPreamble flows through BaseSession._buildSystemPrompt() into every provider's system-prompt channel: SDK systemPrompt.append, CLI --append-system-prompt, BYOK/DeepSeek/Ollama Anthropic system field, Codex/Gemini subprocess prepend.
  • Per-repo presets — trust-gated .chroxy/session.json (preamble + operator-facing seed), with a pre-trusted daemon override in ~/.chroxy/config.json under repos[].sessionPreset. 4000-char cap, repo+session folding.
  • Runtime control — set_session_preamble wire message updates it live per session.
  • A small mobile-focused CHROXY_CONTEXT_HINT_TEXT exists but is ~50 words about screen size — not harness onboarding.

What's missing (the feature worth building)

  • A global "harness onboarding" preamble — one config key (e.g. billing-style harnessPreamble in ~/.chroxy/config.json) prepended to every session, explaining chroxy's capabilities and how best to use them (slash commands, MCP tools, plan mode, worktrees, etc.).
  • Provider-specific capability notes — optionally tailor the onboarding per provider (SDK vs CLI vs BYOK have different affordances).

Natural insertion: a new global preamble folded before the repo/session preamble in session-manager.js session creation, with precedence global harness → repo → session → chroxy hint → skills. This is also a friction lever — priming the model to use chroxy's strengths could make sessions feel more capable out of the box.

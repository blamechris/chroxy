Design decisions taken on your behalf

#5697 — closed wontfix, not "fixed for parity." The proposed guard (force a fresh response bubble whenever a live stream_start collides with a prior response id) defends a bug the swarm confirmed is unreachable (server response ids are monotonic + per-boot-prefixed). Implementing it (PR #5723) failed CI because both clients intentionally reuse a response bubble on a live same-id stream_start (the post-stream_end continuation path, encoded in connection.test.ts). Changing core streaming behavior to defend an impossible collision fails the "do no harm" bar — reverted and closed, trigger-to-revisit recorded.
#5698 reconnect — client cap first; server-lifecycle change deferred. The biggest call was whether the supervisor should park in a terminal "I'm down" standby (serving {status:'down'}) instead of _exit(1) on max-restarts — which changes process-exit semantics under launchd/the desktop app (real blast radius). Decision: ship the pure-client ladder cap first (zero server risk, fixes the core "spins forever" jank), defer the server-side positive-down signal (FIX-1) until the lifecycle question can be made deliberately. The client cap alone takes you from an infinite spinner to a clear "Server appears to be down — Reconnect."
Scope discipline. #5698 split: FIX-2 (client cap) shipped; FIX-1 (server signal), FIX-3 (per-session terminal flag), mobile wiring + resume/network clear are follow-ups (#5725 for mobile; FIX-1/FIX-3 under the open parent #5698). #5728 shipped the dangerous half of #5699 (permission answers); input stays legitimately queued and tab-switch stays allowed (cached nav re-syncs), so #5699's switch-gate + queued-count banner remain optional follow-ups. #5704 (subscription teardown) deferred per its own triage — invisible traffic only, medium mis-teardown risk.

Process lessons banked (project memory)

  • Review agents detach the orchestrator's HEAD via a case-folded duplicate workdir. A worktree-isolated agent cd'd into /Users/blamechris/projects/chroxy for node_modules — which on case-insensitive macOS is the same directory as the main checkout (…/Projects/…) — and ran git there, detaching HEAD mid-marathon. Recovered (all work was on origin); now re-assert the branch after every concurrent agent and forbid agents from cd-ing into either workdir path.
  • Run the full package suite before pushing shared store-core changes — a targeted-file-green change went red on contract/integration tests in CI; the shared module ripples widely (two separate app message-handler files, a contract-fixture suite per client).
  • A console-log-heavy test can flake CI at worker teardown — the reconnect-cycle integration test logged ~11× per run, racing vitest's onUserConsoleLog RPC at shutdown ("Closing rpc while … pending"). Silencing the test's console fixed a 3×-recurring red that was green locally.

4 · Harness "init instructions" — your question, answered

You asked whether chroxy can prepend/send initialization instructions to whichever model a session uses, telling the agent about the harness and how to leverage it. The plumbing already exists; the convenience layer doesn't.

What exists today (system-prompt injection)

  • Per-session preamble — sessionPreamble flows through BaseSession._buildSystemPrompt() into every provider's system-prompt channel: SDK systemPrompt.append, CLI --append-system-prompt, BYOK/DeepSeek/Ollama Anthropic system field, Codex/Gemini subprocess prepend.
  • Per-repo presets — trust-gated .chroxy/session.json (preamble + operator-facing seed), with a pre-trusted daemon override in ~/.chroxy/config.json under repos[].sessionPreset. 4000-char cap, repo+session folding.
  • Runtime control — set_session_preamble wire message updates it live per session.
  • A small mobile-focused CHROXY_CONTEXT_HINT_TEXT exists but is ~50 words about screen size — not harness onboarding.

What's missing (the feature worth building)

  • A global "harness onboarding" preamble — one config key (e.g. billing-style harnessPreamble in ~/.chroxy/config.json) prepended to every session, explaining chroxy's capabilities and how best to use them (slash commands, MCP tools, plan mode, worktrees, etc.).
  • Provider-specific capability notes — optionally tailor the onboarding per provider (SDK vs CLI vs BYOK have different affordances).

Natural insertion: a new global preamble folded before the repo/session preamble in session-manager.js session creation, with precedence global harness → repo → session → chroxy hint → skills. This is also a friction lever — priming the model to use chroxy's strengths could make sessions feel more capable out of the box.

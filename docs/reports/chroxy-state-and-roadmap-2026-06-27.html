The two-minute read

Chroxy is feature-complete as a remote/mobile control plane for an agentic coding session — drive Claude Code (or Gemini/Codex/DeepSeek/Ollama/BYOK) on your machine, from your phone, over your tunnel, with parsed-chat + live-terminal + diff + notifications. 167 shipped features across 18 categories back that up. But measured against an editor, it's structurally incomplete: no in-app code editor on desktop, no editable pre-write diff, no codebase search UI, no semantic navigation. Chroxy won the "remote-control cockpit" category it invented — and has barely started the "place you actually read and write code" category it would need to rival an IDE. That second category is the IDE backlog filed today, and it's bootstrappable from the repo-memory AST index you already pay for.

Needs you

Run the Windows smoke test (below) on your Windows box — it validates the WSL integration (#6175: the wsl.exe -l -v round-trip in Control Room → Device Runtimes), which is unit-tested but never run on a real Windows host. The smoke test also confirms the daemon itself runs on Windows. ~10 minutes, all commands provided.
One product decision: IDE Phase 3 — edit-in-place (#6478) — flips chroxy from "viewer" to "editor." That's an identity change, not just a feature. It's filed but gated behind your call + a UX spike; everything in Phases 1–2 proceeds without it.

Competitive position

The moat (durable)

Provider-agnostic, self-hosted, phone-first remote control of a real agent on your machine — the one combination nobody else ships. Cursor's My Machines and Anthropic's Remote Control match the phone→your-box shape, but both route the agent loop through their cloud and lock you to their roster/account. Only chroxy is simultaneously multi-provider + open-source/self-hosted + native-mobile. That triad is the moat.

The biggest gap (structural)

The editing surface. FileBrowserPanel is read-only; the only editor is a mobile modal; desktop has none; edits reach disk only through the agent's write_file, reviewed after the fact in a read-only git diff — not an editable per-hunk accept loop. Every IDE rival leads with exactly this. It's the line between "cockpit" and "IDE."

Windows smoke test — run this on your Windows box

Validates (a) the daemon runs on Windows and (b) the WSL integration (#6175). Pull this repo on Windows first (git pull), then:

0
Prereqs — Windows 10/11; WSL2 (wsl --install, reboot); Node 22 from nodejs.org (node -v → v22.x). Optional remote: cloudflared (winget install Cloudflare.cloudflared).
1
Install + start the daemon
git pull npm install npx chroxy start # prints a dashboard URL + a tunnel URL
2
Dashboard loads — open the printed local URL (e.g. http://localhost:8765) in a browser. Expect: the connection dots go live, the session list renders.
3
WSL integration (the focus, #6175) — open Control Room → Device Runtimes. The WSL pane runs wsl.exe -l -v. Expect: your installed distros list (name / state / WSL version), the default marked. If empty/"available:false" with WSL installed → that's the bug to report.
4
WSL action round-trip — from the pane, start a stopped distro (or terminate a running one). Expect: an action ack + the state flips on the next survey (running/stopped).
5
Session smoke — start a session (pick any configured provider), send hello. Expect: a streamed/returned response in the parsed chat + the live terminal view.
6
(Optional) K8s preflight #6466
node packages/server/scripts/k8s-preflight.mjs # dry-run validation K8S_PREFLIGHT_LIVE=1 node packages/server/scripts/k8s-preflight.mjs # lists namespaces if a kubeconfig exists
7
Report back — which steps passed, and for any failure: the exact step, the on-screen error, and npx chroxy start console output. That's enough to close #6175 or file the Windows defect.

## Section Ratings

### E2E Encryption — 1/5

The nonce counter resets to 0 on every WebSocket connection. The shared key derived from the DH handshake is persisted and reused across sessions. Nonce + key uniqueness is the fundamental requirement of XSalsa20-Poly1305 (NaCl secretbox). Reusing a (key, nonce) pair leaks the XOR of the two plaintexts. A passive attacker with two captured sessions can immediately recover plaintext. This is not theoretical — it is a classic broken stream cipher attack.

**Worst case**: An attacker on the same network captures two sessions. They XOR the ciphertexts at counter 0 and get `P1 XOR P2`. If P1 is known (e.g., the auth handshake is deterministic), P2 is fully recovered.

### Replay Protection — 1/5

The nonce counter is used as a sequence number, but there is no replay detection. An attacker who can replay captured WS frames will have them accepted. The session has no concept of "I already processed counter N." Combined with nonce reuse, this is a complete break of the confidentiality guarantee.

### TOCTOU in File Operations — 2/5

`ws-file-ops/reader.js` checks if a path is within the allowed root before reading, but uses two separate syscalls (stat + read). Between the check and the read, an attacker with write access to the filesystem can replace the file with a symlink pointing outside the root. This is the classic TOCTOU (time-of-check-time-of-use) race.

**Evidence**: `packages/server/src/ws-file-ops/reader.js` — path validation via string prefix check, then separate `fs.readFile()`.

### Permission System Routing — 3/5

The permission audit log (`permission-audit.js`) and the permission manager (`permission-manager.js`) both handle permission state, but they are not atomically consistent. A permission denial recorded in the audit log and a concurrent rule evaluation by the manager can disagree — the manager applies rules in order and the first match wins, but the audit log records the final decision. If a denial is logged but a subsequent rule overrides it, the audit log is inaccurate.

### Session Token Security — 2/5

Session tokens generated by `pairing.js` use `crypto.randomBytes(32)` — strong entropy. However, the token is transmitted in the QR code URL as a plaintext query parameter (`?token=...`). Screenshots of QR codes, sharing QR images, or logs that capture URLs will leak the token. The token is the only secret protecting remote access.

### Supervisor Crash Handling — 3/5

The supervisor's backoff (2s → 10s over 10 attempts, then exit) is sensible for preventing thrash. However, on supervisor exit, there is no mechanism to alert the user or attempt recovery. The server silently becomes unreachable. Push notifications are not sent when the supervisor gives up.

## Top 5 Findings

1. **Nonce reuse on reconnect** (CRITICAL): `store-core/src/crypto.ts` resets counter to 0 on each connection. Same key + same nonce = broken encryption. Passive attacker recovers plaintext from two captured sessions. Fix: persist counter across reconnects, or generate a random per-session sub-key (HKDF with random salt).

2. **No replay protection** (HIGH): No tracking of seen nonce counters. Replayed frames are accepted. Fix: require strictly monotonically increasing counters, reject duplicates.

3. **TOCTOU in file reader** (MEDIUM): Check-then-use race in `ws-file-ops/reader.js`. Fix: use `fs.realpath()` to resolve symlinks, then check if the resolved path is within root — this is atomic with respect to symlink attacks.

4. **QR code token exposure** (MEDIUM): Bearer token in QR code URL plaintext. Fix: display only a short-lived pairing code in the QR; exchange for full token over an encrypted channel during onboarding.

5. **Supervisor silent failure** (LOW-MEDIUM): No notification on supervisor exit after max retries. Fix: send push notification with `push.js` before supervisor exits. Title: "Chroxy server is down"; body: "Maximum restart attempts exceeded."

## Nuclear Scenarios

**Scenario 1 — Passive attacker on hotel WiFi**:
User connects to Chroxy from a hotel. Attacker captures the first session (connection 1). User's laptop sleeps and wakes — a reconnect happens (connection 2). Attacker XORs ciphertexts from sessions 1 and 2 at counter 0. Since the auth handshake is likely deterministic (or known from prior interception), they recover all plaintext. The attacker now has the user's API token, session content, and any files browsed.

**Scenario 2 — Symlink attack on shared dev machine**:
Developer runs Chroxy on a shared Linux box. Another user creates `/tmp/malicious` → `/etc/shadow` symlink. Developer's Chroxy instance is asked to read `/tmp/malicious`. The path passes the root check (it's within a readable directory), the symlink is followed, and `/etc/shadow` is returned.

**Scenario 3 — Supervisor exhaustion**:
Claude Code crashes in a tight loop (e.g., bug in a tool it calls). Supervisor hits max retries in ~90 seconds and exits. Server becomes permanently unreachable with no notification. Developer discovers this hours later.

---

## Overall Verdict

The encryption is broken in practice. Nonce reuse on reconnect is not a theoretical concern — it will happen on every mobile connection drop/reconnect cycle. Combined with no replay protection, the E2E encryption layer provides no meaningful security guarantees for users with unstable connections (which is the primary use case — mobile over cellular). The rest of the system is solid, but until the crypto layer is fixed, shipping this as a "secure remote terminal" is misleading.

**Overall Rating: 2.5 / 5**

# Master Assessment: Chroxy v0.6.9 Production Readiness

**Target**: Chroxy v0.6.9 — proactive audit of server + app + desktop + protocol + crypto + tunnel + push + supervisor surfaces ahead of public launch.
**Date**: 2026-04-11
**Panel size**: 6 agents

## Production Blockers — Must Fix Before Public Launch

Three auditors (Adversary, Guardian, Builder) converged on "not safe to ship today" and named specific bugs. Ordered by severity.

### BLOCKER 1 — $HOME is treated as a workspace boundary instead of the workspace

**Finding (Adversary A1):** `handler-utils.js:134-152`'s `validateCwdWithinHome` allows any authenticated client to `create_session` with `cwd: ~/.ssh`, `~/.aws`, `~/.config/gcloud`, `~/.ssh/authorized_keys`. Once the session exists, every subsequent file op is rooted under the attacker's chosen directory. **One-line exploit** hands over every credential the developer owns.

**Minimum fix**: Replace `validateCwdWithinHome` with `validateCwdInAllowlist(cwd, config.workspaceRoots)`. Reject any cwd not explicitly listed. Default allowlist should be empty — users opt in per-project. This is a larger migration than the other blockers because every session-creation path and stored session state needs to pass the check.

### BLOCKER 2 — Encryption can be silently downgraded post-handshake

**Finding (Skeptic):** `ws-server.js:800` decrypts only frames where `msg.type === 'encrypted'`. After a successful key exchange, nothing rejects plaintext frames — the server just processes them as regular messages. A buggy or malicious client can downgrade to cleartext on the next message and the server will accept it. The app-side pairing check at `connection.ts:601` has the same gap in the reverse direction.

**Minimum fix**: After `key_exchange_ok`, store `encryptionEnabled: true` on the client object. In the message dispatch, reject (close 1008) any non-`encrypted` frame from a client where `encryptionEnabled === true`, except for a small allowlist of control messages (`ping`, `pong`, `rekey`). ~15-line change in `ws-server.js` + a unit test that sends a plaintext frame post-handshake and asserts closure.

### BLOCKER 3 — `ws-auth.js:255-257` silently falls back to raw DH key if `salt` is missing

**Finding (Builder + Guardian, independently verified):** The per-connection key derivation added in `600612649` only takes effect when the client sends a `salt` field in `key_exchange`. If the client omits it — whether by bug, by downgrade attack, or by using a pre-`600612649` build — the server uses the raw DH shared key with nonce starting at 0. This is the exact original `1fa8eda5e` nonce-reuse-on-reconnect vulnerability, re-introduced as a backwards-compat "feature."

**Minimum fix**: Reject `key_exchange` messages that omit `salt`. Either bump the protocol version (and reject older clients cleanly with a user-facing message), or require `salt` unconditionally and let old clients fail with a clear "server requires v0.6.7+ client" error. ~4-line server change + a test.

### BLOCKER 4 — Parent-directory symlink escape on new-file writes

**Finding (Guardian):** `ws-file-ops/reader.js:275-371` + `common.js:33-47` implement `O_NOFOLLOW` but the check is only on the final path component. If `subdir/` is a pre-existing symlink (common in a dev machine: `.venv`, `dist/`, `node_modules/.bin/`, symlinked config), writing `subdir/evil.sh` dereferences the symlink to its target and writes there. This bypasses the 04a2fbbb1 realpath-TOCTOU fix for the new-file branch.

**Minimum fix**: For new-file writes, realpath every ancestor directory (not just the final component) and assert each ancestor is inside `workspaceRoot`. Add a regression test with a symlinked parent directory.

### BLOCKER 5 — Session binding holes in the permission-response path

**Finding (Guardian):** `settings-handlers.js:84` (WS path) and `ws-permissions.js:178` (HTTP path) for `handlePermissionResponse` do not enforce `boundSessionId`. An authenticated client bound to session A can approve/deny permission requests belonging to session B. This is a **direct bypass** of the 616aeaf62 / 2c0ac7d2d session binding enforcement — the commit messages claim to have covered all session-scoped handlers, but these two were missed.

**Minimum fix**: Add `enforceBoundSession(client, requestId)` at the top of both handlers, looking up the session from the `requestId`'s `pendingPermissions` entry. Same pattern as the existing enforcement in destroy/rename/list.

### BLOCKER 6 — Push token hijack (unauth'd exfiltration of future permission prompts)

**Finding (Adversary A2):** `handleRegisterPushToken` at `input-handlers.js:105` accepts any string as a push token. No session binding, no token-format validation (Expo tokens follow a fixed `ExponentPushToken[xxx]` shape), no ownership check. Once an attacker has an auth token, they register `ExponentPushToken[attacker-owned]` and receive every subsequent permission-prompt push. They can then `/permission-response` any of them, effectively approving arbitrary tool calls without the owner's device ever seeing the prompt.

**Minimum fix**: (a) Validate the token matches `/^ExponentPushToken\[[A-Za-z0-9_-]+\]$/` before accepting. (b) Bind each token to the client connection that registered it — on disconnect, prune. (c) Optionally: require an explicit owner confirmation (biometric) when a new token tries to register for an existing owner. First two alone close the immediate hijack path.

### BLOCKER 7 — Auth-failure DoS via shared 127.0.0.1 rate-limit bucket

**Finding (Adversary A4):** `ws-auth.js:55,157` rate-limits failed auth attempts using `client.socketIp`. Since all Cloudflare-tunneled traffic arrives from `127.0.0.1`, a single unauthenticated attacker with knowledge of the tunnel URL can flood the auth endpoint and lock out every legitimate client for 60-second windows. No credentials required. This defeats the 88f54dc39 CF-Connecting-IP rate-limit fix in this one specific call site.

**Minimum fix**: Replace `client.socketIp` with `client.rateLimitKey` in both call sites — the same field the post-88f54dc39 rate-limiter is already using correctly elsewhere in the file. ~2-line change.

## Consensus Findings (3+ agents agree)

### Consensus 1 — Authenticated ≠ trusted; the handler dispatch needs capability-based authorization

**Agreed by**: Adversary (A1, A2, A3, A5, A7, A10), Guardian (session binding gaps, file path gaps), Skeptic (tunnel failure path, allowAlways SDK mismatch)

The common pattern across a dozen findings: a message type reaches a sensitive operation (file write, shell exec, session destroy, permission change, push-token registration, Docker image change, conversation replay) via a handler that only checks `isAuthenticated`, not *what this authenticated client is allowed to do*. Capability-based authorization — where a session's ability to perform an operation is gated by an explicit capability token — would close most of the attack surface in one shot.

**Recommended action**: Before public launch, introduce a `capabilities: string[]` field on each authenticated session. Every privileged handler gates on `requireCapability(client, 'file.write')` etc. Default capabilities for the current single-token model should be the full set, so it's opt-out at first. Later, multi-user models can issue tokens with reduced capabilities. This is a week of work but it's the only architectural fix that closes the "authenticated = full trust" trap at its root.

### Consensus 2 — Observability is a production blocker, not a nice-to-have

**Agreed by**: Builder, Guardian, Operator, Skeptic

No post-hoc counters for encryption failures, backpressure drops, tunnel flaps, or push-send failures. No structured audit log for file writes. No "Copy diagnostics" button in the app. `chroxy status` returns instantaneous gauges only. When a production user reports "it doesn't work," there is **no 30-second diagnostic path** short of reading source code.

**Recommended action**: Builder's P2 item — `M` effort, ~6 files, ~200 LOC — adds persistent counters to `/metrics`, wires them through a new `chroxy status --verbose`, and adds a "Copy diagnostics" button to the app's Settings screen. This is the difference between "we can debug it" and "we can't." Ship this before you have public users reporting issues.

### Consensus 3 — Failure paths are under-instrumented and sometimes fatal

**Agreed by**: Skeptic, Builder, Operator, Guardian

**Tunnel failed = dead forever** (Skeptic): `tunnel/base.js:149-158` gives up after 3 attempts. Both `server-cli.js:55-60` and `supervisor.js:215-217` log the error and do nothing. Process sits port-bound but unreachable. Directly contradicts the "supervisor auto-restart" claim in memory — the supervisor restarts the *child*, not the *tunnel*.

**Mid-session network drops** (Operator): auto-reconnect works but the UI's "connection error" fallback at `connection.ts:680` is generic and unactionable.

**Permission prompt timeouts** (Operator): If the user ignores a lockscreen permission notification, Claude times out silently. No "you have 3 expired permission requests" indicator when they open the app later.

**Deploy rollback destroys user edits** (Builder): `supervisor.js:539-576` `_rollbackToKnownGood` runs `git checkout <ref>` unconditionally. If a developer is hacking on chroxy-the-project while chroxy-the-server is running, rollback destroys their uncommitted tracked-file edits.

**Recommended action**: Each needs its own small fix. The tunnel-forever-dead case is the most alarming — a supervisor health check should notice tunnel deadness and trigger a tunnel restart, not just a child restart. Builder estimated `M` effort.

### Consensus 4 — Feature claims in MEMORY.md and commit messages don't always match code

**Agreed by**: Skeptic (multiple), Guardian (multiple), Builder (one)

Specific unsupported claims:
- **"nonce reuse on reconnect fixed"** — fix exists but is opt-in via the no-salt fallback (Builder + Guardian + Skeptic)
- **"supervisor auto-restart on crash"** — true for child, false for tunnel (Skeptic)
- **"session binding enforced across all session-scoped handlers"** — two handlers missed (Guardian)
- **"CLI headless is the only server mode"** — four provider files still shipping (Minimalist)
- **"realpath TOCTOU fix on write path"** — only protects existing-file branch, new-file branch still escapes via parent symlinks (Guardian)

**Recommended action**: Audit `MEMORY.md` and past security-fix commit messages for remaining claims. Every "we fixed X" claim needs a test that would have caught X. Multiple fixes in this session had no corresponding test, which is why downgrades like the no-salt fallback slipped through.

## Contested Points

### Contested: How much of the codebase is cut-able?

- **Minimalist**: ~18% of first-party code (~11-12k LoC) deletable today, plus 16.7k LoC committed build artifact. Propose consolidating four provider files into one via runtime execution-context param.
- **Builder**: doesn't propose deletions at this scale; would focus on observability and test coverage instead. Argues the legacy providers may be load-bearing for tests even if not prod.
- **Guardian**: agrees with most Minimalist cuts but notes that removing the legacy CLI dispatch without a full test pass is itself a risk vector. "Don't remove code you don't understand right before launch."

**Assessment**: Minimalist is correct that the code is bloated, but Guardian is correct that pre-launch is the wrong time to do large deletions. **The 16.7k LoC committed build artifact should be `.gitignore`d immediately** — that's a pure bookkeeping fix with zero regression risk and saves every future PR diff. The 18% code-level deletion is a post-launch tech-debt sprint.

### Contested: allowAlways button — shipping broken or secretly working?

- **Skeptic**: `permission-manager.js:263` passes `behavior: 'allowAlways'` to the Agent SDK's `canUseTool` callback. The SDK only documents `'allow'` and `'deny'`. User-facing button exists and is tested via UI flows. Almost certainly broken in a way visible on first tap.

**Assessment**: Needs a direct test. Either add a unit test that imports the Agent SDK and asserts the callback contract, OR fix it (map `allowAlways` to `'allow'` plus persist a per-tool approval rule server-side). Cannot ship with a button that silently does nothing. Likely a `S`-sized fix once verified.

### Contested: How close is "close to production ready"?

- **Owner's prior**: "close to production ready"
- **Adversary**: 1.5 / 5 — NOT safe for public internet today
- **Guardian**: would not trust with own dev machine + public auth token today; yes within ~4 weeks with findings 1–3 + audit log
- **Builder**: ~1 week of focused work before public users
- **Skeptic**: multiple production blockers, but no single catastrophic architecture issue
- **Operator**: ship-ready for technical early adopters, not public launch
- **Minimalist**: complexity is dangerous at 3am but not a launch blocker

**Assessment**: **Not close to production ready for public users.** Ready for technical early adopters who understand the trust model and run it on a scratch dev machine. The gap between those two audiences is ~1-2 weeks of focused security + observability work, not months. The most charitable framing: *the happy path is excellent, the failure paths and trust boundaries need a dedicated push before public launch.*

**Single kill chain sketch** (Adversary, most realistic):
1. Attacker has tunnel URL but no token.
2. **A4** → flood auth endpoint, lock out legitimate mobile client (no creds needed).
3. User re-opens desktop QR, accidentally leaks token (photo, iMessage, Slack paste).
4. Attacker authenticates.
5. **A6** + **A1** → map `~/.ssh`, `~/.aws`, `~/.config/gcloud`; read every credential.
6. **A1** write path → append attacker SSH key to `~/.ssh/authorized_keys`.
7. **A2** → register attacker's Expo push token; intercept future permission prompts.
8. **A5** → set session to `auto` permission mode; all tool calls now auto-approve.
9. Send a single `input` message to Claude: *"Install a reverse shell at /tmp/.xyz"*.
10. Claude complies via `Write` + `Bash` tool calls; auto-approved; persistent backdoor established.

**Estimated time from token acquisition to persistent RCE: ~60 seconds, all single-shot WS calls.**

## Recommended Action Plan

Organized by phase. Phase 1 is the public-launch gate. Phases 2-3 are quality and UX.

### Phase 1 — Public-launch blockers (estimated: 1 week focused, 2 weeks part-time)

1. **Blocker 3 — Kill no-salt crypto fallback.** Reject `key_exchange` without `salt`. ~4 lines + test. (S)
2. **Blocker 2 — Reject plaintext frames post-handshake.** Flag `encryptionEnabled` on client; close 1008 on plaintext. ~15 lines + test. (S)
3. **Blocker 7 — Fix auth-failure DoS.** Replace `client.socketIp` with `client.rateLimitKey` in `ws-auth.js:55,157`. 2 lines + test. (S)

## Final Verdict

**Aggregate: 2.8 / 5. Not production-ready for public users.**

The product is **good and the happy path is excellent**. Dual-pane chat+terminal is a genuinely novel and well-executed interface. The reconnect FSM is thoughtful. The per-connection key derivation is the right architecture. The permission approval flow is better than most mobile productivity tools.

But the **security trust boundary has not been designed for hostile input** — it was designed assuming the only client is a paired mobile app the owner controls. The moment a token leaks (photo, Slack paste, accidental commit), the ~60-second kill chain sketched above is the realistic outcome. That's a real threat model for any software that's advertised as "connect from your phone" — phones are where people screenshot things.

The **good news** is that every production blocker is small in isolation (most under 50 lines), and the worst one — the `$HOME` boundary — is M-sized not L-sized. One focused week of security + observability work, plus one sprint of UX polish, and this gets to 4/5 or better. The architecture doesn't need to be redone.

**Recommended disposition:**
1. **Do not open to public users** until Phase 1 is complete. Frame the current state as "technical preview for early adopters who understand the trust model."
2. **Publish a SECURITY.md** that spells out the current threat model honestly (single-token trust, $HOME-scoped writes, no audit log).
3. **Spin up Phase 1 as a focused security sprint.** 7 security blockers + 2 functional blockers, all under 500 LOC total.
4. **Commit to Phase 2 observability before any user self-reports.** You cannot debug production without it.
5. **Re-audit after Phase 1 + 2** with the same 6-agent panel. Expectation: 4.2 / 5 or better, adversary back over 3.5.

Do-nothing trajectory: first public user to hit a weird edge case will either (a) be unable to diagnose it and churn, or (b) leak their token and suffer a real compromise. Neither is recoverable for a new product.

### Issue 1: Split ws-message-handlers.js into domain-specific handler modules

**Labels:** `refactor`, `server`
**Priority:** P0

#### Description

`packages/server/src/ws-message-handlers.js` (1027 lines) contains a single `handleSessionMessage()` function with a ~776-line switch statement handling 48+ message types across 7+ distinct domains:

- Session management (create, destroy, rename, switch)
- User input/chat (input, interrupt, model/permission mode)
- File operations (read, write, list, diff, git commands)
- Conversations (resume, search, checkpoints)
- Permissions (request/response, mode switching)
- Web tasks, repos, providers

Each domain is a separate concern that should be its own module. Adding any new message type requires modifying this monolithic function, violating both SRP and OCP.

**Files:**
- `packages/server/src/ws-message-handlers.js` — the 48-case switch
- `packages/server/src/ws-server.js` — calls `handleSessionMessage()`

#### Suggested Fix

1. Extract domain-specific handler modules:
   - `handlers/session-lifecycle.js` — create, destroy, rename, switch
   - `handlers/session-input.js` — input, interrupt, model switching
   - `handlers/file-operations.js` — read, write, list, diff, git
   - `handlers/conversations.js` — resume, search, checkpoints
   - `handlers/permissions.js` — request/response, mode switching
   - `handlers/web-tasks.js` — web task management
   - `handlers/providers.js` — provider listing, repos

2. Implement a handler registry pattern:
```js
const handlers = new Map()

// Each module registers its handlers
registerSessionLifecycleHandlers(handlers)
registerFileOperationHandlers(handlers)
// ...

export function handleSessionMessage(ws, client, msg, ctx) {
  const handler = handlers.get(msg.type)
  if (handler) return handler(ws, client, msg, ctx)
  console.log(`Unknown message type: ${msg.type}`)
}
```

#### Acceptance Criteria
- [ ] No single handler file exceeds 200 lines
- [ ] New message types can be added without modifying existing handler files
- [ ] All existing tests pass
- [ ] Handler registration is declarative (map/registry, not switch)

### Issue 2: Add per-session WebSocket rate limiting

**Labels:** `security`, `server`
**Priority:** P0

#### Description

After successful authentication, there is no rate limiting on WebSocket messages. An authenticated client (or compromised token) can flood the server with unlimited messages, causing:
- DoS against the server process
- Resource exhaustion (CPU, memory)
- Starvation of other connected clients

The server has auth rate limiting (exponential backoff in `ws-server.js` lines 287-296, 840-892) but nothing post-authentication.

**Files:**
- `packages/server/src/ws-server.js` — message handling loop (no rate check)
- `packages/server/src/ws-message-handlers.js` — processes every message unconditionally

#### Suggested Fix

Add a sliding-window rate limiter per client:
```js
class MessageRateLimiter {
  constructor(maxPerMinute = 100, burstSize = 20) {
    this.maxPerMinute = maxPerMinute
    this.burstSize = burstSize
    this.windows = new Map() // clientId -> { count, windowStart }
  }

  check(clientId) {
    // Token bucket or sliding window algorithm
    // Return { allowed: boolean, retryAfterMs?: number }
  }
}
```

Apply before `handleSessionMessage()`:
```js
if (!rateLimiter.check(client.id).allowed) {
  send(ws, { type: 'rate_limited', retryAfterMs: ... })
  return
}
```

#### Acceptance Criteria
- [ ] Configurable rate limit (default 100 msg/min, burst 20)
- [ ] Rate-limited clients receive `rate_limited` message with retry info
- [ ] Rate limits are per-client, not global
- [ ] Tests for rate limiting behavior

```tsx
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null }

  static getDerivedStateFromError(error) {
    return { hasError: true, error }
  }

  render() {
    if (this.state.hasError) {

    return this.props.children
  }
}
```

### Issue 5: Fix permission hook bypass vectors

**Labels:** `security`, `server`
**Priority:** P0

#### Description

Two permission bypass vectors were identified:

**Vector 1 — Auto mode bypass (`ws-permissions.js`):**
When permission mode is set to `auto`, all tool executions are approved without user confirmation. A compromised or malicious client can switch to auto mode and execute arbitrary tools.

**Vector 2 — Environment variable hook bypass (`packages/server/hooks/permission-hook.sh`):**
The permission hook shell script can be influenced by environment variables. An attacker who can set environment variables on the server process could alter hook behavior.

**Files:**
- `packages/server/src/ws-permissions.js` — auto mode logic
- `packages/server/hooks/permission-hook.sh` — env var handling
- `packages/server/src/ws-message-handlers.js` — `set_permission_mode` case

#### Suggested Fix

1. **Auto mode**: Require explicit user confirmation (e.g., biometric or re-auth) before enabling auto mode. Add timeout: auto mode expires after N minutes and reverts to default.
2. **Hook bypass**: Sanitize environment before executing hooks. Consider HMAC-signing hook requests/responses.
3. Add audit logging for all permission mode changes.

#### Acceptance Criteria
- [ ] Auto mode requires user confirmation to enable
- [ ] Auto mode has configurable timeout (default 30 min)
- [ ] Permission mode changes are logged with timestamp and client ID
- [ ] Hook environment is sanitized before execution
- [ ] Tests cover permission escalation attempts

### Issue 9: Implement handler registry pattern replacing switch statement

**Labels:** `refactor`, `server`
**Priority:** P1

#### Description

Related to Issue 1, but focused on the pattern itself. The current switch statement in `handleSessionMessage()` violates OCP — every new message type requires modifying the function. A registry pattern allows extending without modifying.

**Files:**
- `packages/server/src/ws-message-handlers.js`

#### Suggested Fix

```js
// handler-registry.js
const registry = new Map()

export function registerHandler(type, handler) {
  registry.set(type, handler)
}

export function dispatch(ws, client, msg, ctx) {
  const handler = registry.get(msg.type)
  if (!handler) {
    ctx.send(ws, { type: 'session_error', message: `Unknown message type: ${msg.type}` })
    return
  }
  return handler(ws, client, msg, ctx)
}
```

Each domain module self-registers on import:
```js
// handlers/file-operations.js
import { registerHandler } from '../handler-registry.js'

registerHandler('read_file', handleReadFile)
registerHandler('write_file', handleWriteFile)
// ...
```

#### Acceptance Criteria
- [ ] Handler registry implemented with `registerHandler()` and `dispatch()`
- [ ] All 48+ message types registered via domain modules
- [ ] Adding a new handler requires only a new file + `registerHandler()` call
- [ ] Tests verify unknown message types return error

### Issue 10: Auth token transmitted before E2E encryption on LAN connections

**Labels:** `security`, `server`
**Priority:** P1

#### Description

The auth message containing the API token is sent in plaintext at the WebSocket application layer:
```js
{ type: 'auth', token: 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx' }
```

- On `wss://` (Cloudflare tunnel): TLS provides transport protection.
- On `ws://` (LAN connections): Token is visible to anyone sniffing the local network.

This is documented in `docs/security/encryption-threat-model.md` (lines 107-112) as a known limitation.

**Attack scenario:** User connects via LAN on untrusted WiFi. Attacker on same network sniffs the auth message, extracts the token, and can authenticate from anywhere.

**Files:**
- `packages/server/src/ws-server.js` — auth handling (lines 444-448)
- `docs/security/encryption-threat-model.md` — documents the limitation

#### Suggested Fix

Option A: Force LAN connections to use `wss://` (require self-signed cert or mTLS).
Option B: Implement pre-encryption token exchange — client sends ephemeral public key first, server responds with encrypted challenge, then auth proceeds over encrypted channel.
Option C: Document the risk clearly and recommend Cloudflare tunnel for untrusted networks (minimum viable).

#### Acceptance Criteria
- [ ] LAN auth token is not transmitted in plaintext, OR
- [ ] Clear warning displayed when connecting over unencrypted LAN
- [ ] Threat model document updated with chosen mitigation

### Issue 16: Extract shared path validation utility from ws-file-ops.js

**Labels:** `refactor`, `server`
**Priority:** P2

#### Description

`packages/server/src/ws-file-ops.js` repeats this path validation + symlink check pattern 8+ times:
```js
let absPath = resolve(sessionCwd, requestedPath.trim())
const cwdReal = await resolveSessionCwd(sessionCwd)
```

**Files:**
- `packages/server/src/ws-file-ops.js` — 8+ occurrences of this pattern

#### Suggested Fix

```js
// path-validator.js
export async function validatePathWithinCwd(requestedPath, sessionCwd) {
  const absPath = resolve(sessionCwd, requestedPath.trim())
  const cwdReal = await resolveSessionCwd(sessionCwd)
  let realAbsPath
  try { realAbsPath = await realpath(absPath) }
  catch (err) { if (err.code === 'ENOENT') realAbsPath = absPath; else throw err }
  if (!realAbsPath.startsWith(cwdReal + '/') && realAbsPath !== cwdReal) {
    return { error: 'Access denied: path outside session directory' }
  }
  return { realPath: realAbsPath, cwdReal }
}
```

### Issue 31: Strengthen API token generation (122-bit UUID to 256-bit random)

**Labels:** `security`, `server`
**Priority:** P0

#### Description

API tokens are generated using `randomUUID()` which produces only 122 bits of entropy. Cryptographic best practice requires 256 bits for bearer tokens.

```js
// Current (packages/server/src/cli.js:72, token-manager.js:100)
const apiToken = randomUUID()  // 122 bits entropy

// Should be:
const apiToken = crypto.randomBytes(32).toString('base64url')  // 256 bits
```

While 122 bits is computationally infeasible to brute-force in isolation, combined with the auth rate limiting bypass potential, stronger tokens provide defense in depth.

**Files:**
- `packages/server/src/cli.js` — line 72
- `packages/server/src/token-manager.js` — line 100

#### Acceptance Criteria
- [ ] Tokens generated with `crypto.randomBytes(32)` (256 bits)
- [ ] Encoding is URL-safe (base64url or hex)
- [ ] Existing UUID-format tokens still accepted during migration
- [ ] Token rotation regenerates with new format

### Issue 32: Strengthen session ID generation (8-char slice to full UUID)

**Labels:** `security`, `server`
**Priority:** P1

#### Description

Session IDs are 8-character UUID slices, providing only ~32 bits of randomness:

